1. RGB LED indicator

A single onboard WS2812 LED on GPIO 8 communicates the device state through color (state) and pattern (urgency / activity). The Matter OnOff cluster on Endpoint 1 acts as a brightness gate: OnOff = false turns the LED off without changing the underlying state; OnOff = true resumes the indicator.

Boot PULSE

White breathing pulse, 1.5 s period. Shown for the first 5 s after every reset.

Commissioning 2 Hz

Blue blink, 250 ms on / 250 ms off. Pairing window open or device not yet commissioned.

Thread detached 0.5 Hz

Yellow slow pulse. Commissioned but not currently attached to the Thread mesh.

Running SOLID

Green solid. Commissioned and Thread attached. Normal operation.

Btn preview — open commissioning 4 Hz

Blue fast blink while BOOT is held past 7 s. Release now to open the commissioning window.

Btn preview — factory reset ~6 Hz

Red fast blink while BOOT is held past 15 s. Release now to factory-reset the device.

Factory reset active 5 Hz

Red blink. The reset has been scheduled and the device will reboot shortly.

2. BOOT button (GPIO 9)

Three actions are distinguished by hold duration. The Matter Contact Sensor on Endpoint 3 reports the raw debounced press/release state regardless of duration.

< 1 s

Short press

Contact Sensor reports pressed → released. No other side effect. Any release before the 7 s threshold leaves device state unchanged.

7–15 s

Open commissioning window

Calls OpenBasicCommissioningWindow(180 s). The device starts BLE-advertising again so an additional Matter admin can pair. Existing fabric, Thread credentials, and node ID are preserved.

≥ 15 s

Factory reset

Calls Server::ScheduleFactoryReset(). All Matter fabrics, ACLs, bindings, and Thread credentials are wiped. The device reboots into the un-commissioned state.

Tip. Hold thresholds are visualized live: at 7 s the LED switches to fast blue blink, at 15 s to fast red blink. Release at any moment triggers the action whose threshold was last crossed (or none, for a release before 7 s).

3. When to use what

  • Lost the device in matter-server, but Thread is healthy — hold BOOT 7–15 s, then re-pair from matter-server via QR. No need to re-flash or reset Thread state.
  • Adding the node to a second fabric (multi-admin, e.g. Apple Home + matter-server) — hold BOOT 7–15 s, then commission from the second controller.
  • Migrating to a different lab / different Thread network / unrecoverable state — hold BOOT ≥ 15 s for full factory reset.
  • Brand-new node, never commissioned — indicator is already blue-blinking out of the box; commission from matter-server using the QR code in the boot log.

5. Lab onboarding (Stage 1)

The next stage-1 step is a per-device onboarding manifest written into runtime/matter-node-onboarding/<serial>.json during the lab provisioning flow. Real per-device manifests are runtime state and must not be committed to Git.

The production direction is a factory-data provisioning flow: build one firmware image, generate unique per-device Matter credentials, write them to factory data / NVS at provisioning time, store the QR in a registry, and commission by QR.

6. Boot diagnostics & heartbeat

At every reset, before Matter starts, the firmware prints:

  • Firmware version string (v6.1.3).
  • esp_reset_reason() decoded — distinguishes power-on from watchdog/panic.
  • OpenThread platform configuration installed.
  • Matter endpoint IDs for light / temperature / contact sensor.
  • BLE QR payload and manual pairing code (test setup).

After Matter starts, every 30 s a heartbeat line is logged:

Heartbeat: thread_attached=<0|1> commissioned=<0|1> window_open=<0|1>
           free_heap=<bytes> uptime_s=<seconds>

This is the fastest way to confirm the device is alive without opening ot-ctl on the border router.

7. Build & flash

Source the local ESP-IDF environment first:

source /home/ets/.espressif/v5.4.1/esp-idf/export.sh
cd /home/ets/bms-et-sensors/nodes/esp32c6zero/matter-node

Standard build / flash:

idf.py set-target esp32c6
idf.py build
idf.py -p /dev/ttyACM1 flash

Optional clean rebuild:

rm -rf build sdkconfig managed_components dependencies.lock
idf.py build
Shared USB hub warning. OTBR should use a stable /dev/serial/by-id/... path for the SONOFF dongle, so simple ttyACM0/ttyACM1 renumbering is no longer the main risk. The real failure mode on this host is plugging or resetting an ESP32-C6 on the same upstream USB hub branch: that can briefly re-enumerate USB devices or dip bus power, and OTBR may lose the SONOFF RCP. Prefer separate physical USB ports or a powered hub. If OTBR drops after flashing or reconnecting an ESP32-C6, rerun ./scripts/restart-openthread.sh.
